# 需求输入源处理指南

## 飞书 PRD 文档

### 读取方式
使用 Feishu MCP 工具读取文档内容：
- 通过文档 token 或链接获取文档
- 使用 `mcp_lark-mcp_docx_v1_document_rawContent` 获取纯文本内容
- 使用 `mcp_lark-mcp_docx_builtin_search` 搜索文档内容

### 解析策略
1. **文档结构识别**
   - 识别标题层级（H1, H2, H3）
   - 识别列表和表格
   - 识别代码块和引用

2. **关键信息提取**
   - 功能需求：查找"功能"、"需求"、"特性"等关键词
   - 业务规则：查找"规则"、"约束"、"限制"等关键词
   - 交互要求：查找"交互"、"流程"、"操作"等关键词
   - 数据需求：查找"数据"、"字段"、"接口"等关键词

3. **结构化整理**
   - 将提取的信息按分析框架组织
   - 补充缺失的信息点
   - 确认模糊的描述

### 常见 PRD 结构
- **背景和目标**：说明为什么需要这个功能
- **功能描述**：详细描述功能点
- **用户故事**：从用户角度描述需求
- **交互设计**：描述用户操作流程
- **数据需求**：描述需要的数据和接口
- **验收标准**：描述功能完成的判断标准

## 手动输入需求

### 引导式问询
当用户手动输入需求时，使用结构化问询引导：

1. **功能概述**
   - 功能名称是什么？
   - 功能要解决什么问题？
   - 目标用户是谁？

2. **功能点**
   - 主要功能有哪些？
   - 每个功能的具体描述？
   - 功能的优先级如何？

3. **交互流程**
   - 用户如何使用这个功能？
   - 操作步骤是什么？
   - 有哪些异常情况需要处理？

4. **数据需求**
   - 需要哪些数据？
   - 数据从哪里来（API、用户输入等）？
   - 数据如何展示？

5. **特殊要求**
   - 是否有性能要求？
   - 是否有兼容性要求？
   - 是否有安全性要求？

### 需求澄清
- 对模糊的描述进行追问
- 对缺失的信息进行补充
- 对矛盾的需求进行确认

## 文件输入需求

### 支持的文件格式
- Markdown (.md)
- 文本文件 (.txt)
- Word 文档 (.docx) - 需要先转换为文本

### 解析策略
1. **读取文件内容**
2. **识别文档结构**（类似 PRD 解析）
3. **提取关键信息**
4. **结构化整理**

### 常见文档结构
- 需求文档：通常包含功能描述、交互设计、数据需求
- 用户故事：通常包含角色、目标、场景
- 技术文档：通常包含架构设计、接口规范、数据结构
