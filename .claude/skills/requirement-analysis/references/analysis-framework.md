# 五维度分析框架

## 维度一：业务场景分析

**目标**：理解需求的业务背景和使用场景。

分析要点：
- **用户角色**：哪些角色会使用这个功能？各角色的操作权限是什么？
- **使用频率**：高频操作还是低频操作？影响 UI 设计优先级
- **业务规则**：有哪些业务约束？如审批流程、状态限制
- **场景关系**：场景间是否有依赖、互斥或顺序关系？

输出格式：
```markdown
### 业务场景

| 场景 | 角色 | 触发条件 | 预期结果 |
|------|------|----------|----------|
| 场景1 | 管理员 | 点击xx按钮 | 显示xx列表 |
```

## 维度二：功能点拆解

**目标**：将需求拆解为可开发的功能单元。

分析要点：
- **CRUD 操作**：是否涉及增删改查？各操作的具体规则
- **搜索/筛选**：支持哪些筛选维度？是否需要模糊搜索？
- **批量操作**：是否需要批量选择、批量处理？
- **导入/导出**：是否需要 Excel/CSV 导入导出？模板格式？
- **权限控制**：功能级别的权限粒度

输出格式：
```markdown
### 功能清单

#### 模块A
- [ ] 功能1：描述（优先级：P0/P1/P2）
  - 规则：xxx
  - 边界条件：xxx
```

## 维度三：交互流程梳理

**目标**：理清用户操作路径和页面流转。

分析要点：
- **正常流程**：主路径的操作步骤
- **异常流程**：错误提示、空状态、网络异常处理
- **页面跳转**：页面间的导航关系
- **组件交互**：弹窗、抽屉、Toast 等交互方式
- **表单流程**：表单校验规则、提交流程、草稿保存

输出格式：
```markdown
### 操作流程

1. 用户进入xx页面
2. 点击「新建」按钮 → 打开新建弹窗
3. 填写表单 → 校验
   - 成功：提交并关闭弹窗，刷新列表
   - 失败：显示字段级错误提示
```

## 维度四：数据模型分析

**目标**：梳理前端需要处理的数据结构。

分析要点：
- **核心实体**：主要数据对象的字段定义
- **接口映射**：前端字段与后端 API 字段的对应关系
- **状态枚举**：所有状态值及其含义
- **关联关系**：实体间的一对多、多对多关系
- **字段校验**：必填、格式、长度等校验规则

输出格式：
```typescript
interface Order {
  id: string
  status: OrderStatus
  items: OrderItem[]
  createdAt: string
  // ...
}

enum OrderStatus {
  Draft = 'draft',
  Pending = 'pending',
  Completed = 'completed',
}
```

## 维度五：技术实现评估

**目标**：评估实现难度和技术选型。

分析要点：
- **新增/复用**：哪些组件可复用，哪些需要新开发？
- **复杂度评估**：各功能的技术复杂度（低/中/高）
- **技术风险**：性能瓶颈、兼容性问题、第三方依赖风险
- **依赖关系**：功能间的开发依赖顺序
- **工作量估算**：按人天给出粗略估算

输出格式：
```markdown
### 技术评估

| 功能 | 复杂度 | 可复用组件 | 新增组件 | 风险 |
|------|--------|------------|----------|------|
| xx列表 | 低 | Table, SearchBar | - | 无 |
| xx表单 | 高 | Form | DynamicFields | 动态表单逻辑复杂 |
```
