# 需求分析

## 工作流程

1. **获取需求输入** — 从 PRD 文档、产品描述或设计稿获取需求，按来源选择处理方式。详见 [references/input-sources.md](references/input-sources.md)。
2. **需求结构化分析** — 按五维度分析：业务场景、功能点、交互流程、数据模型、技术实现。详见 [references/analysis-framework.md](references/analysis-framework.md)。
3. **生成分析文档** — 输出结构化需求分析文档（见下方输出格式）。
4. **生成 Todolist** — 按功能模块/优先级/开发阶段组织，任务含动作动词+对象+可选验收标准，标注依赖。详见 [references/todolist-template.md](references/todolist-template.md)。

## 分析原则

- **结构化**：使用统一五维度框架，不遗漏关键维度
- **完整性**：挖掘细节，识别边界条件与异常流程
- **可执行**：分析文档与 Todolist 清晰、具体、可直接指导开发
- **可追溯**：保持需求来源与任务的关联性

## 处理规则

- **删除线内容**：需求文档中带删除线的文字视为已删除功能，不纳入开发计划；仅在分析文档末尾以「已删除/已下线需求」备注
- **范围控制**：仅规划本项目内需要开发的内容，不规划非本项目范围的功能（如后端接口、第三方系统）
- **模块对应**：从需求中提取模块/子模块，与项目中的模块名称、代码路径对应；若找不到匹配模块且无法判断是否为新增需求，须与用户确认后再写入分析文档

## 输出格式

### 需求分析文档

```markdown
# 需求分析文档：[功能名称]

## 1. 需求概述
- 背景与目标
- 涉及的用户角色
- 核心价值

## 2. 业务场景分析
- 主要业务场景描述
- 场景间的关系与依赖

## 3. 功能点拆解
- 按模块组织的功能清单
- 每个功能点的输入/输出/规则

## 4. 交互流程梳理
- 主要操作流程（正常流 + 异常流）
- 页面间跳转关系
- 状态变化

## 5. 数据模型分析
- 核心数据结构
- 接口字段映射
- 状态枚举

## 6. 技术实现评估
- 技术选型建议
- 复杂度评估
- 风险识别
```

### Todolist

按需求选择组织方式（功能模块/优先级/开发阶段），任务写法与粒度标注见 [references/todolist-template.md](references/todolist-template.md)。

# Todolist 模板与规范

## 组织方式

根据需求规模选择组织方式：

### 按功能模块（推荐）

```markdown
## 模块A：用户管理

### 基础 CRUD
- [ ] [P0] 实现用户列表页（Table + 分页 + 搜索）
- [ ] [P0] 实现新建用户弹窗（表单校验 + 提交）
- [ ] [P0] 实现编辑用户（复用新建表单，预填数据）
- [ ] [P1] 实现删除用户（确认弹窗 + 批量删除）

### 高级功能
- [ ] [P1] 实现用户角色分配（多选 + 树形结构）
- [ ] [P2] 实现用户导入导出（Excel 模板 + 进度反馈）

## 模块B：权限管理
- [ ] ...
```

### 按优先级

```markdown
## P0 — 核心功能（本迭代必须完成）
- [ ] 实现xx功能
- [ ] 实现xx功能

## P1 — 重要功能（本迭代尽量完成）
- [ ] 实现xx功能

## P2 — 优化项（下一迭代）
- [ ] 优化xx体验
```

### 按开发阶段

```markdown
## 阶段一：基础框架（第 1-2 天）
- [ ] 搭建页面路由结构
- [ ] 定义 TypeScript 类型

## 阶段二：核心功能（第 3-5 天）
- [ ] 实现主要 CRUD
- [ ] 对接后端 API

## 阶段三：完善体验（第 6-7 天）
- [ ] 添加 loading/empty 状态
- [ ] 错误处理与提示
```

## 任务写法规范

### 格式

```
- [ ] [优先级] 动作动词 + 对象 + 可选验收标准
  - 依赖：依赖的其他任务
  - 备注：技术细节或注意事项
```

### 动作动词

| 动词 | 含义 | 示例 |
|------|------|------|
| 实现 | 从零开发 | 实现用户列表页 |
| 添加 | 在已有基础上增加 | 添加搜索筛选功能 |
| 修改 | 改动已有逻辑 | 修改状态流转规则 |
| 优化 | 改善性能/体验 | 优化列表加载性能 |
| 重构 | 结构性调整 | 重构表单校验逻辑 |
| 对接 | 连接外部 | 对接用户列表 API |
| 配置 | 设置/调整配置 | 配置路由权限守卫 |

### 粒度控制

- **太粗**：❌ 实现用户管理模块
- **太细**：❌ 给按钮添加 onClick 事件
- **合适**：✅ 实现用户列表页（Table + 分页 + 搜索筛选）

原则：每个任务 **0.5~2 天** 的工作量，可独立验证完成。

### 依赖标注

```markdown
- [ ] [P0] 定义 User/Role TypeScript 类型
- [ ] [P0] 实现用户列表 API 封装
  - 依赖：TypeScript 类型定义
- [ ] [P0] 实现用户列表页 UI
  - 依赖：API 封装、TypeScript 类型
```

# 需求输入来源处理

## PRD 文档

### Markdown / 文本文件
- 直接读取文件内容
- 识别标题层级结构
- 提取需求描述、功能清单、原型说明
- 注意删除线内容（`~~文字~~`）视为已废弃

### 在线文档链接
- 提示用户将内容粘贴到对话中
- 或者导出为 Markdown/PDF 后作为文件输入

### 图片/设计稿
- 识别页面布局和交互元素
- 提取标注的文字、按钮、表单等
- 判断是 B 端（管理后台）还是 C 端（用户端）页面
- 注意区分当前页面属于哪个模块

## 口头描述

用户直接描述需求时：

1. **提炼核心功能**：从描述中识别主要功能点
2. **追问关键细节**：
   - 用户角色有哪些？
   - 核心操作流程是什么？
   - 有没有现成的设计稿或 PRD？
   - 是新功能还是在现有模块上修改？
3. **确认范围**：明确哪些在本次开发范围内
4. **结构化整理**：将口头信息按五维度框架整理

## 已有代码上下文

当在现有项目上开发时：

1. 用 `analyze_project` 工具分析项目结构
2. 定位需求涉及的模块和文件
3. 理解现有的数据模型和 API 结构
4. 评估改动影响范围

## 输入预处理

无论来源，预处理步骤：

1. **去噪**：去除格式标记、页眉页脚、无关内容
2. **结构化**：将输入按功能模块初步分组
3. **标记不确定项**：标记需要确认的模糊需求
4. **提取关键词**：识别技术术语、业务概念、模块名称

# 五维度分析框架

## 维度一：业务场景分析

**目标**：理解需求的业务背景和使用场景。

分析要点：
- **用户角色**：哪些角色会使用这个功能？各角色的操作权限是什么？
- **使用频率**：高频操作还是低频操作？影响 UI 设计优先级
- **业务规则**：有哪些业务约束？如审批流程、状态限制
- **场景关系**：场景间是否有依赖、互斥或顺序关系？

输出格式：
```markdown
### 业务场景

| 场景 | 角色 | 触发条件 | 预期结果 |
|------|------|----------|----------|
| 场景1 | 管理员 | 点击xx按钮 | 显示xx列表 |
```

## 维度二：功能点拆解

**目标**：将需求拆解为可开发的功能单元。

分析要点：
- **CRUD 操作**：是否涉及增删改查？各操作的具体规则
- **搜索/筛选**：支持哪些筛选维度？是否需要模糊搜索？
- **批量操作**：是否需要批量选择、批量处理？
- **导入/导出**：是否需要 Excel/CSV 导入导出？模板格式？
- **权限控制**：功能级别的权限粒度

输出格式：
```markdown
### 功能清单

#### 模块A
- [ ] 功能1：描述（优先级：P0/P1/P2）
  - 规则：xxx
  - 边界条件：xxx
```

## 维度三：交互流程梳理

**目标**：理清用户操作路径和页面流转。

分析要点：
- **正常流程**：主路径的操作步骤
- **异常流程**：错误提示、空状态、网络异常处理
- **页面跳转**：页面间的导航关系
- **组件交互**：弹窗、抽屉、Toast 等交互方式
- **表单流程**：表单校验规则、提交流程、草稿保存

输出格式：
```markdown
### 操作流程

1. 用户进入xx页面
2. 点击「新建」按钮 → 打开新建弹窗
3. 填写表单 → 校验
   - 成功：提交并关闭弹窗，刷新列表
   - 失败：显示字段级错误提示
```

## 维度四：数据模型分析

**目标**：梳理前端需要处理的数据结构。

分析要点：
- **核心实体**：主要数据对象的字段定义
- **接口映射**：前端字段与后端 API 字段的对应关系
- **状态枚举**：所有状态值及其含义
- **关联关系**：实体间的一对多、多对多关系
- **字段校验**：必填、格式、长度等校验规则

输出格式：
```typescript
interface Order {
  id: string
  status: OrderStatus
  items: OrderItem[]
  createdAt: string
  // ...
}

enum OrderStatus {
  Draft = 'draft',
  Pending = 'pending',
  Completed = 'completed',
}
```

## 维度五：技术实现评估

**目标**：评估实现难度和技术选型。

分析要点：
- **新增/复用**：哪些组件可复用，哪些需要新开发？
- **复杂度评估**：各功能的技术复杂度（低/中/高）
- **技术风险**：性能瓶颈、兼容性问题、第三方依赖风险
- **依赖关系**：功能间的开发依赖顺序
- **工作量估算**：按人天给出粗略估算

输出格式：
```markdown
### 技术评估

| 功能 | 复杂度 | 可复用组件 | 新增组件 | 风险 |
|------|--------|------------|----------|------|
| xx列表 | 低 | Table, SearchBar | - | 无 |
| xx表单 | 高 | Form | DynamicFields | 动态表单逻辑复杂 |
```
